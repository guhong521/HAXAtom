# HAXAtom 项目计划书

## 一款「原子化解耦、乐高式拼装」的全栈AI智能体管理与多端分发平台

***

## 一、项目核心设计思想

本项目的核心灵魂是\*\*「彻底解耦、池化复用、预设组合、轻量化交付」\*\*，打破传统Bot项目“配置与实例强绑定、修改成本高、复用性差”的痛点，打造一套高自由度、高扩展性、低维护成本的AI智能体架构，核心理念拆解为5大不可动摇的设计原则：

1. **【单一职责·彻底解耦】**
   将AI智能体的核心能力要素完全拆分解耦，模型配置、人设提示词、插件工具、知识库、记忆策略五大核心模块完全独立，各自实现单一职责。修改任一模块的配置不会对其他模块产生任何影响，从根源上解决配置耦合带来的维护难题。
2. **【原子化池化·无限复用】**
   所有解耦后的能力要素均以「原子化资源」的形式存入对应资源池，形成**模型池、提示词池、插件池、知识库池、记忆池**五大独立资源池。一次配置即可全项目无限复用，无需重复编写配置，大幅降低冗余代码与重复工作量。
3. **【乐高式拼装·预设组合】**
   以「**预设方案（Preset）**」为核心载体，用户仅需在预设方案中从五大资源池中挑选对应的原子化资源，即可像搭乐高一样快速拼装出一个完整的AI智能体实例。预设方案仅做资源引用与组合规则定义，不存储冗余配置，实现“一键切换智能体、一套配置多场景复用”。
4. **【轻量化部署·重度化能力】**
   采用单体架构设计，无需复杂中间件与分布式部署，前端与后端深度整合，支持单exe可执行文件、Docker镜像一键启动，极致降低部署门槛；同时原生集成企业级RAG知识库、多端机器人接入、插件化工具调用、可视化配置管理等重度能力，兼顾易用性与功能性。
5. **【一套配置·全渠道触达】**
   采用「统一配置中心+多渠道适配层」架构，一套预设方案配置，可同时无缝同步到Web对话端、飞书/钉钉/QQ/Telegram等多IM渠道，无需为不同渠道重复配置，实现一次配置、全平台生效。

一套配置，全端触达，无限自定义的AI智能体中枢

## 四、核心架构设计

整体采用**分层解耦架构**，以「统一配置中心+预设组合引擎」为核心中枢，所有模块围绕核心设计思想搭建，逻辑清晰，扩展性极强。

### 整体分层架构（从下到上）

```
┌─────────────────────────────────────────────────────────────────┐
│ 交互层：Web管理端(Bot视图)、Web对话端(Chat视图)、多IM渠道接入端 │
├─────────────────────────────────────────────────────────────────┤
│ 路由适配层：统一消息封装、鉴权限流、全局异常拦截、多渠道适配     │
├─────────────────────────────────────────────────────────────────┤
│ 核心引擎层：预设组合引擎、LangChain对话中枢、RAG检索引擎、工具调用引擎 │
├─────────────────────────────────────────────────────────────────┤
│ 资源池层：五大原子化资源池 + 统一配置中心                        │
├─────────────────────────────────────────────────────────────────┤
│ 存储层：SQLite结构化存储、ChromaDB向量存储、本地文件系统         │
└─────────────────────────────────────────────────────────────────┘
```

### 各层核心职责（贴合核心设计思想）

1. **存储层**：项目的底层数据底座
   - SQLite：分表存储五大资源池的原子化配置、预设方案配置、对话历史、用户信息，支持配置的持久化与快速查询
   - ChromaDB：分集合存储不同知识库的向量数据，与知识库池配置一一对应，支持数据持久化
   - 本地文件系统：支持配置文件的导入导出、备份恢复，满足用户用配置文件管理的核心需求
2. **资源池层**：项目的核心配置底座，完全遵循「彻底解耦、池化复用」理念
   - 统一配置中心：管理五大独立的原子化资源池，提供标准化的增删改查、导入导出、加密解密能力
   - 五大资源池：模型池、提示词池、插件池、知识库池、记忆池，所有池完全独立，互不依赖
3. **核心引擎层**：项目的大脑中枢，完全遵循「乐高式拼装、预设组合」理念，**基于LangChain最新标准实现**
   - **预设组合引擎**：核心中的核心，基于**LangGraph**工作流框架，根据预设方案配置从五大资源池拉取原子化资源，动态构建状态图（StateGraph）实现智能体编排
   - **LCEL对话链**：基于**LCEL声明式语法**（`|`操作符组合Runnable），实现提示词模板、模型调用、输出解析的链式组装，完全替代已废弃的LLMChain/SequentialChain
   - **RAG检索引擎**：对接知识库池配置，基于LCEL实现`检索→重排序→上下文注入`的检索增强链，支持文档向量化与检索结果引用
   - **工具调用引擎**：对接插件池配置，基于LangGraph的`ToolNode`实现Agent的动态工具调用，支持工具执行结果回传与循环决策
   - **记忆管理**：基于**LangGraph持久化层**或LCEL的`RunnableWithMessageHistory`，实现对话历史的自动存储与注入，替代已废弃的Memory类
4. **可观测性层（LangSmith集成）**：项目的调试与优化底座，实现全链路可观测
   - **全链路追踪**：所有LCEL链与LangGraph工作流自动接入LangSmith，可视化展示调用链路、输入输出、耗时分布
   - **RAG检索调试**：记录每次检索的查询语句、召回文档、重排序结果，精准定位「检索不准」问题
   - **提示词版本管理**：Prompt池的每次修改自动记录版本，支持A/B测试与回滚
   - **Token成本监控**：按预设方案、按模型统计Token用量与API成本，优化模型选择
   - **用户反馈收集**：对话消息的点赞/点踩数据自动关联到LangSmith Run，用于效果评估与迭代
   - **性能瓶颈分析**：识别慢查询、高频调用、错误率高的环节，指导优化方向
5. **路由适配层**：项目的统一入口，完全遵循「一套配置·全渠道触达」理念
   - 统一消息封装：无论请求来自Web端还是第三方IM渠道，全部封装为内部标准消息格式，走同一套核心逻辑
   - 多渠道适配层：新增渠道仅需实现对应适配器，无需修改核心配置与引擎逻辑，扩展性极强
   - 鉴权限流：实现管理员鉴权、接口频率限制、全局异常捕获、日志记录，保障项目安全
6. **交互层**：项目的用户交互界面，完全遵循「配置与视图分离」理念
   - Bot视图（管理端）：配置的「生产者/编辑器」，负责五大资源池的管理、预设方案的编辑与管理，所有操作写入统一配置中心
   - Chat视图（消费端）：配置的「使用者」，对话时直接读取预设方案的组合配置，调用核心引擎能力，与管理端共用完全同一套底层配置
   - 多IM渠道端：飞书/钉钉/QQ/Telegram等渠道的机器人，复用同一套预设方案配置，实现全渠道触达

## 五、核心配置体系详解

本章节为项目核心落地规范，完全实现「配置完全解耦、配置文件管理、高自定义、一键切换」的核心需求。

### 1. 五大原子化资源池规范

所有资源池均采用「唯一ID标识、独立存储、标准化结构、可复用、可导入导出」的设计原则，每个资源池的配置均可单独导出为YAML/JSON文件。

#### （1）模型配置池 (Model Pool)

存储所有大模型的独立配置，每个模型为一个原子化条目，一次配置，全预设方案复用。
**标准化配置结构（YAML示例）**：

```yaml
model_id: model_deepseek_v3_chat
model_name: DeepSeek-V3 对话模型
model_type: chat
provider: deepseek
api_base: https://api.deepseek.com
api_key: ENCRYPTED(sk-xxxxxx) # 加密存储，禁止明文
default_params:
  temperature: 0.7
  top_p: 0.9
  max_tokens: 4096
  frequency_penalty: 0
is_active: true
create_time: 2026-04-01
update_time: 2026-04-01
```

#### （2）提示词配置池 (Prompt Pool)

存储所有人设、系统提示词的独立配置，每个提示词为一个原子化条目，一次配置，全预设方案复用。
**标准化配置结构（YAML示例）**：

```yaml

prompt_name: 傲娇女友人设v2

  1.  称呼用户为“笨蛋”，但会主动关心用户的日常
  2.  不会直接说情话，只会用别扭的方式表达关心
  3.  禁止出现机械、生硬的AI式回复
variables:
  - user_name
  - meet_date
temperature_override: null
is_active: true
create_time: 2026-04-01
update_time: 2026-04-01
```

#### （3）插件配置池 (Plugin Pool)

### 2. 核心预设方案（Preset）规范

预设方案是「乐高式拼装」的核心载体，仅做资源引用与组合规则定义，不存储任何冗余的原子化配置，每个预设方案可单独导出为YAML/JSON文件，支持一键分享、一键导入、一键切换。

**标准化预设方案结构（YAML示例）**：

```yaml
# 预设方案唯一标识
preset_id: preset_tsundere_girlfriend_deepseek
# 预设方案名称（前端展示用）
preset_name: 傲娇女友（DeepSeek-V3）
# 方案描述
description: 基于DeepSeek-V3模型的傲娇女友聊天Bot，开启联网搜索能力
# 核心资源引用：仅填对应资源池的唯一ID，不存储冗余配置
selected_model: model_deepseek_v3_chat
selected_prompt: prompt_tsundere_girlfriend_v2
selected_plugins:
  - plugin_web_search_baidu
  - plugin_weather_query
selected_knowledge_bases:
  - kb_our_chat_history
selected_memory: memory_short_term_10_rounds
# 可选：方案专属覆盖配置，仅对当前方案生效，不修改资源池的默认配置
overrides:
  model_params:
    temperature: 0.9
    top_p: 0.95
# 渠道配置：一套方案，全渠道生效
channel_config:
  enable_web: true
  enable_feishu: true
  enable_telegram: false
# 元数据
is_active: true
create_time: 2026-04-01
update_time: 2026-04-01
```

### 3. 预设方案加载核心逻辑

当用户在前端选择一个预设方案时，后端执行标准化加载流程，全程无硬编码，完全动态组装：

1. 读取预设方案配置，校验所有引用的资源ID是否存在、是否可用
2. 从五大资源池中，拉取对应ID的完整原子化配置
3. 应用预设方案中的`overrides`覆盖配置，生成当前方案专属的运行参数
4. 将所有配置传入LangChain对话中枢，动态组装成完整的AI智能体运行实例
5. 对话过程中，所有能力均基于当前预设方案的配置执行，与其他方案完全隔离

## 六、功能模块与交互设计

### 1. Bot视图（管理端）：配置生产与管理中心

核心定位是五大资源池与预设方案的可视化编辑器，所有操作最终写入统一配置中心，与Chat视图完全共享底层数据。
核心功能模块：

- **资源池管理中心**：模型管理、提示词管理、插件管理、知识库管理、记忆管理五大子页面，支持配置的新增、编辑、删除、导入、导出、批量操作
- **预设方案管理中心**：预设方案的创建、编辑、复制、导入、导出、启用/禁用，可视化下拉选择资源池的原子化配置，一键拼装智能体
- **多端接入管理**：各IM渠道的Webhook配置、接入状态监控、消息日志查看，复用预设方案配置
- **系统管理**：管理员权限、数据备份与恢复、日志查看、系统配置

### 2. Chat视图（消费端）：配置使用与对话中心

核心定位是预设方案的使用者，与管理端共用完全同一套底层配置，管理端修改配置后，对话端实时生效。
核心功能模块：

- **预设方案一键切换器**：对话窗口顶部的下拉选择框，一键切换不同的预设方案，毫秒级变更Bot的人设、模型、能力
- **沉浸式对话窗口**：支持Markdown解析、代码高亮与复制、流式打字机效果、中断响应、对话清空/导出
- **对话历史管理**：按预设方案分类存储对话历史，不同方案的对话完全隔离，互不干扰
- **快捷操作入口**：支持拖拽文件直接上传至知识库、快速调整当前方案的模型参数、快捷开启/关闭插件

## 九、开发与安全规范

### 1. 代码开发规范

- 后端Python代码严格遵循PEP8规范，使用Black自动格式化，isort整理导入顺序，全量类型注解覆盖
- 前端Vue代码严格遵循Vue3 Composition API官方风格指南，使用TypeScript类型注解，Eslint+Prettier自动格式化，组件化拆分，逻辑与视图分离
- 所有变量、函数、文件命名语义化，核心逻辑必须有注释说明，禁止拼音缩写与无意义命名
- 所有接口必须遵循RESTful规范，入参校验全覆盖，统一响应格式，统一错误码体系

### 2. 配置管理规范

- 所有原子化配置必须有唯一ID，结构标准化，禁止非结构化配置
- 敏感配置（API Key、密码）必须加密存储，禁止明文写入配置文件与数据库
- 配置修改必须有版本记录，支持回滚，避免误操作导致配置丢失
- 配置文件导入导出必须做格式校验与安全校验，禁止恶意配置导入

### 3. 安全规范

- 所有管理接口必须经过JWT鉴权，禁止未授权访问
- 所有对话接口、文件上传接口必须实现频率限制，防止恶意刷接口
- 第三方插件必须实现权限隔离，禁止执行高危系统命令、访问敏感配置
- 支持HTTPS配置，禁止公网环境下HTTP明文传输敏感信息
- 实现数据一键备份与恢复功能，避免用户数据丢失

## 十、部署与交付方案

### 1. Windows单exe交付（核心轻量化方案）

- 实现思路：
  1. 前端构建为dist静态资源文件夹，内嵌到Python项目中
  2. PyInstaller打包Python环境、所有依赖、前端静态资源、数据库驱动全部打包进单个exe文件
  3. 优化打包配置，解决路径问题、依赖缺失问题，实现双击exe即可启动服务，自动打开浏览器访问控制台
- 优势：用户无需安装Python环境、无需配置任何依赖，开箱即用，极致降低使用门槛

### 2. Docker一键部署（服务器部署方案）

- 实现思路：
  1. 多阶段构建Dockerfile，第一阶段构建前端项目，第二阶段搭建Python后端环境，整合前端静态资源
  2. 编写docker-compose.yml，配置端口映射、数据卷挂载，实现配置、数据库、知识库的持久化
  3. 支持环境变量注入敏感配置，无需修改镜像即可调整参数
- 优势：一条命令即可完成部署，支持Linux服务器、NAS等环境，数据持久化，维护方便
